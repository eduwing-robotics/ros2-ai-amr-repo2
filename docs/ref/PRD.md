# PRD

## 한 줄 소개

Unity Digital Twin 관제 화면에서 **TurtleBot 2대가 박물관/미술관의 액자형 중요물품을 순찰·보호하고 이상 상황을 감지·확인하는 디지털트윈경비로봇** 시스템을 보여주는 프로젝트입니다.

## 왜 만들까?

야간 실내 경비와 화재·침입 대응을 자동화하기 위해 한 대의 로봇이 모든 일을 다 하는 대신, **순찰 로봇과 확인 로봇이 분업·협력**하는 모습을 사람이 한눈에 이해할 수 있게 보여주기 위해서입니다. 발표 컨셉은 박물관/미술관이며, 로봇은 사진이 부착된 액자형 목표물을 카메라로 인식하고 보호 대상으로 다룹니다.

## 누가 쓸까?

- 로봇·자율주행·디지털트윈·다중 로봇 협업을 배우는 학생과 개발자
- 실내 경비·재난 대응 로봇을 연구하는 사람 (소방, 보안, 미탐사지역)
- 로봇 관제 시스템과 이벤트 기반 대응 로직을 설계하는 사람
- 로봇 시연을 보는 일반 관람자

## 이번에 만들 것

- 듀얼 TurtleBot3 (**`tb3_1` 티원** 순찰/감지/비전(D435), **`tb3_2` 젠지** 출동/확인/센서(Pi Camera IMX219 + Arduino 4종)) 시나리오. 호스트: 티원=`t1@192.168.0.250` (hostname `rb`) / 젠지=`urhynix-robot` (kim@192.168.0.82). DECISION-LOG 2026-06-02 참조
- TurtleBot3 SLAM/Nav2 기반 순찰 주행
- **아두이노 센서 4종 [정본 v18 갱신]**: PIR(사람), 온도(화재 감지), 소리(이상 소음), 레이저+워터펌프(화재 대응 분사). [이전: 조도, 불꽃→현재 제외. 핀 매핑은 제안으로 정본 v18엔 미명시]
- **이벤트 발행 → Unity 관제 표시 → tb3_2 출동 → Pi Camera 확인 → DB 기록**의 전체 파이프라인
- Unity Digital Twin 관제 UI (지도, 로봇 상태, 이벤트 마커, 카메라 패널, 운영 대시보드)
- **박물관/미술관 액자 보호 시나리오**: 액자형 사진 타깃을 `protected_asset`으로 등록하고, 카메라가 중요물품 존재/이상 여부를 확인
- 영상 라이브 스트리밍 (tb3_2 → Unity 관제)
- 이벤트/대응 로그 DB 저장
- 이동 좌표 로그, 사진, 짧은 영상, 사운드 클립을 DB 메타데이터와 Storage 경로로 저장
- 데이터 라벨링 + AI 분류 (오탐/실탐 구분 보조)

## 이번에는 안 만들 것

- FR5 로봇팔 제어 및 픽앤플레이스
- 실시간 사람 추적 (실내 좌표 기반 follow)
- 실제 화재 테스트 (안전상 모의 입력으로 대체)
- 완전 자동 보안 시스템 수준의 인증/권한
- 복잡한 멀티로봇 최적 경로 계획 (waypoint 기반 단순 출동만)
- 로봇 간 물리적 협동 작업 (push/lift 등)

## 성공 기준

- **v1**: tb3_1이 SLAM/Nav2로 사전 정의된 waypoint 루프를 안정적으로 순찰한다.
- **v2**: 아두이노 센서 1종(PIR 또는 조도) 이벤트가 ROS-TCP로 발행되고 Unity 관제 화면에 위치·이벤트가 실시간 마커로 표시된다.
- **v3**: 이벤트가 DB(`events` 테이블)에 저장되고, tb3_2가 감지 좌표 근처 waypoint로 (시뮬레이션 또는 실기) 출동한다. Pi Camera 캡처가 `camera_captures`에 저장된다.
- **v4**: 시나리오 4종(야간/PIR/소리/화재 모의)을 박물관/미술관 액자 보호 컨셉으로 시연하고, 발표용 지표(이벤트 감지 횟수, 평균 출동 시간, 확인 성공률, 오탐률)를 표로 보여준다.
- **확장 성공 기준**: 카메라가 액자형 사진 타깃을 중요물품으로 인식하고, 이벤트 전후 이동 좌표·사진·영상·사운드 저장 경로가 DB에서 한 세션으로 조회된다.
- **발표 기준**: 일반인이 "1호기가 무엇을 보고, 어떤 이벤트를 발행했고, 2호기가 어떻게 출동해 확인했는지"를 화면만 보고 이해할 수 있다.

## MVP 한 줄

> tb3_1이 박물관/미술관 구역을 순찰하며 액자형 중요물품과 외부자 이벤트를 감지하면 Unity 관제 화면에 위치·이벤트가 표시되고, tb3_2가 감지 지점으로 출동해 카메라로 확인하며, 이동 좌표·사진·영상·사운드와 대응 결과를 DB에 기록한다.

## 데이터 수집 전략

단순 시연이 아니라 **이벤트 기반 다중 로봇 대응 데이터셋**을 함께 수집한다. 발표·취업 포트폴리오 핵심 무기.

| 필드 | 설명 |
|---|---|
| `session_id` | 한 번의 시연/순찰 세션 |
| `robot_id` | `tb3_1`, `tb3_2` |
| `timestamp` | 이벤트/pose 기록 시각 (UTC, ms) |
| `x`, `y`, `theta` | 로봇 pose |
| `event_type` | 현재 DB: `dark`, `pir`, `noise`, `fire` / 확장 예정: `asset_seen`, `asset_missing`, `camera_confirm` |
| `severity` | 위험도 (0~3) |
| `image_path` | 카메라 캡처 저장 경로 |
| `video_path` | 확장 예정: 이벤트 전후 짧은 영상 저장 경로 |
| `audio_path` | 확장 예정: 소리/상황음 저장 경로 |
| `protected_asset_id` | 확장 예정: 액자형 사진 타깃 또는 중요물품 ID |
| `response_time` | 이벤트 발생 → tb3_2 도착까지 시간(초) |
| `result` | `confirmed`, `false_alarm`, `missed`, `unverified` |

세션 다회 누적이 발표 핵심. 가능하면 같은 트랙에서 시나리오를 반복해 데이터 양을 확보한다.

## 발표 포인트

1. **"한 대가 아닌 두 대의 분업 협력"** — 감지 로봇 + 출동 로봇 역할 분리를 디지털트윈으로 시각화.
2. **"이벤트 기반 대응 데이터셋"** — 감지→출동→확인 흐름을 정량 지표로 제시 (감지률·평균 출동 시간·확인 성공률·오탐률).
3. **"박물관/미술관 컨셉"** — 사진이 붙은 액자형 타깃을 중요물품으로 등록하고 카메라 확인 결과를 남긴다.
4. **"안전 우선 설계"** — 화재 테스트는 모의 입력으로 대체. LiDAR/Nav2가 최종 안전 판단.
5. **"확장 가능한 구조"** — 센서 추가, 로봇 추가, 시나리오 추가가 모두 메시지 계약 변경만으로 가능.

## 하드웨어 구성 (확정)

- **로봇 모델**: TurtleBot3 **Burger**
- **MCU**: **Arduino Uno R3** (아두이노 기본 키트)
- **센서 부**: 미니 브레드보드 + 점퍼선 + 센서 4종(PIR · 온도 · 소리 · 레이저) + 릴레이 모듈(워터펌프) [정본 v18: LDR·불꽃→온도·레이저·워터펌프 교체]
- **데이터 경로**: 센서 → 브레드보드 공통 5V/GND → Arduino → **USB Type-B 케이블** → Raspberry Pi → `urhynix_sensor_bridge` → `/security/event`
- **전원 경로**: **OpenCR 5V 핀 → Arduino 5V 핀** (점퍼 2줄, 5V + GND). USB도 동시에 꽂혀 있지만 Uno 내부 P-MOSFET이 자동 선택. AA 배터리 등 별도 전원 **추가하지 않음**.
- **적층 구조 (Burger, 한 단 추가 X)**:
  1. LDS LiDAR (**최상단** — 360° 시야 양보 X)
  2. Raspberry Pi (한쪽으로 치우쳐 마운트) + **반대편 빈 공간에 Arduino + 미니 브레드보드 양면테이프 부착** ← NEW 배치
  3. OpenCR (5V 핀에서 Arduino로 점퍼 분기)
  4. 배터리 / 모터
- **핀 할당** [제안 매핑]: PIR → D2 · 온도 → A0 · 소리 → D3 · 레이저 → D4 · 릴레이(워터펌프) → D5 [정본 v18에는 핀 번호 미명시, 구 슬롯 재사용 추정]
- **전류 마진**: LDS 400mA + OpenCR 자체 100mA + Arduino+센서 150mA = ~650mA (OpenCR 5V 한계 ~1A 안)
