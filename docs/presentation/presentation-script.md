# URHYNIX 발표 대본

> 사용법: 화면(빔)에는 **키워드 슬라이드**, 손에는 이 대본. 슬라이드 번호와 아래 번호가 1:1로 맞아요.
> 굵은 줄이 그 슬라이드에 떠 있는 키워드, 그 아래가 말로 할 설명입니다.

---

## ① 프로젝트 개요

**1. 표지 — URHYNIX, 박물관을 지키는 디지털 트윈 경비 로봇**
로봇 두 대가 밤의 전시장을 돌면서 침입과 화재, 도난을 스스로 살핍니다. 그 모습을 화면(디지털 트윈)으로 똑같이 따라 봅니다. TurtleBot3 두 대, ROS2와 Unity, 박물관·미술관 야간 경비가 주제고, 저희는 로봇 2조입니다.

**2. 왜 만들었나**
넓은 전시장을 사람이 밤새 지키긴 힘듭니다. 그 일을 로봇에게 맡깁니다.
- 넓은 공간을 24시간 — 전시장 전체를 쉬지 않고 돌며 살펴야 하는데, 사람은 지칩니다.
- 상황마다 다른 대응 — 침입, 화재, 작품 도난, 관람객 접촉. 상황이 다르면 대응도 달라야 합니다.
- 혼자가 아니라 둘이서 — 한 대가 힘이 빠지면 다른 한 대가 순찰을 이어받습니다.

**3. 두 대의 로봇, 서로 다른 역할**
눈이 좋은 로봇과 감각이 예민한 로봇, 둘이 짝을 이룹니다.
- 티원(T1) — 눈이 좋은 로봇입니다. RealSense 깊이 카메라로 작품과 사람을 자세히 봅니다. 비전·물체 인식(YOLO) 담당.
- 젠지(GenG) — 감각이 예민한 로봇입니다. 라즈베리파이 카메라와 여러 센서로 이상을 먼저 알아챕니다. 센서·순찰·출동 담당.

**4. 전체 그림 한 장 (사진)**
로봇이 본 것이 화면과 데이터베이스까지 어떻게 이어지는지 한눈에 보여주는 구성도입니다. 로봇 → ROS2 → Unity 관제 화면 → 데이터베이스로 흐릅니다.

**5. 보여줄 5가지 상황**
발표에서 시연할 시나리오입니다.
- ① 침입자 감지 — 폐관 후 사람이 보이면 노랑 → 빨강 단계로 확인하고 112 신고를 띄웁니다.
- ② 작품 분실 — 기준 사진과 비교해 작품이 사라지면 다시 확인하고 알립니다.
- ③ 화재 의심 — 온도가 오르면 곧장 빨강. 119 신고와 함께 물 펌프를 분사합니다.
- ④ 관람객 접촉 — 개장 중 작품에 손이 닿으면 음성으로 안내합니다.
- ⑤ 배터리 인계 — 힘이 빠진 로봇은 남은 순찰을 친구에게 넘기고 충전하러 갑니다.

**6. 팀 — 로봇 2조**
넷이 나눠 맡았습니다. 김주영은 기획·데이터베이스·시간/위치 기록, 김선일은 Unity 관제 화면·요구사항 정리, 임현찬은 로봇 주행·센서 연결, 박태진은 카메라·통합·발표 준비를 맡았습니다.

## ② 시스템 구조 — 데이터가 화면까지

**1. 표지 — 로봇은 어떻게 Unity 화면에 나타날까?**
센서·카메라·위치 데이터가 로봇에서 출발해 관제 UI에 그려지기까지의 한 흐름을 설명합니다. 키워드는 ROS2, ROS-TCP-Endpoint, Unity ROS-TCP-Connector, 그리고 Domain ID 210입니다.

**2. 큰 그림 — 3단계로 끝**
어려워 보여도 데이터는 항상 같은 길을 지나요.
1. 로봇이 내보낸다 — 로봇이 자신의 위치, 라이다, 카메라, 배터리, 센서값을 ROS2 토픽으로 내보냅니다.
2. 가운데 다리가 받는다 — ROS-TCP-Endpoint라는 중간 프로그램이 TCP(포트 10000)로 메시지를 Unity에 전달합니다.
3. Unity가 받아서 그린다 — Unity의 Subscriber가 토픽을 받아 지도, 카메라, 센서 카드 같은 화면에 표시합니다.

**3. 전체 데이터 경로 (다이어그램)**
로봇 하드웨어에서 관제 화면까지의 경로입니다. 로봇 HW(TurtleBot3·LiDAR·카메라·OpenCR·Arduino) → ROS2 노드(bringup·cartographer·sensor_bridge) → ROS-TCP-Endpoint(TCP 10000 다리) → Unity의 ROS-TCP-Connector → 토픽별 C# Subscriber → UI View(지도·카메라·센서 카드) 순입니다. 같은 네트워크에서 통신하려면 모든 노드가 같은 Domain ID(210)와 같은 RMW를 써야 합니다.

**4. 주요 토픽 계약 (CONTRACT.md §1)**
Unity가 구독하는 핵심 토픽입니다. 토픽 이름이 조금만 틀려도 양쪽이 다 깨지므로 계약서처럼 정해뒀습니다. `/tb3_1/pose`(PoseStamped, 10Hz, 로봇 위치 마커), `/map`(OccupancyGrid, 1Hz, 2D 지도 배경), `/tf`(TFMessage, 30Hz+, 좌표 계산), 카메라 이미지(15Hz), 배터리(1Hz), `/security/event`(단발, 이벤트 마커), `/goal_pose`(단발, Unity→로봇 출동 명령)입니다.

**5. 예시로 보는 구독 흐름 (flow)**
토픽에서 메시지를 받아 Subscriber가 화면에 표시하는 과정입니다. 카메라는 `/tb3_2/camera/image_raw` → Image → CameraStreamSubscriber.cs → 라이브 패널, 배터리는 `/tb3_1/battery_state` → BatteryState → BatterySubscriber.cs → 상단바 %, 위치는 `/tf`와 `/pose` → TFMessage → RobotPoseSubscriber.cs → 지도 마커로 갑니다.

**6. Unity 구독자/발행자 모음**
`Assets/Scripts/Ros/` 폴더에 토픽마다 스크립트가 하나씩 있습니다. RobotPoseSubscriber(위치 계산), MapSubscriber(지도 이미지), CameraStreamSubscriber(JPEG 디코딩), BatterySubscriber(전압→%), PirSubscriber(동작 감지 판정), SoundSubscriber(소음 swing), TemperatureSubscriber(온도 raw·NTC 1Hz), LaserSubscriber(레이저 송신부 상태·현재 미결선), DispatchPublisher(우클릭 시 `/goal_pose` 발행). 조도(LuxSubscriber)는 정본 v18에서 폐기돼 빠졌습니다.

**7. 반대 방향 — Unity가 명령을 보낸다 (다이어그램)**
한쪽 방향만이 아니라, 관제자가 지도를 우클릭하면 로봇에게 출동 명령이 날아갑니다. 지도 우클릭 → DispatchPublisher → `/goal_pose`(목표 좌표) → Nav2(경로 생성·주행) → 로봇 이동 순입니다.

**8. 왜 환경을 맞춰야 할까**
통신이 안 될 때 원인의 90%가 이 세 가지입니다.
- Domain ID = 210 — 숫자가 다르면 같은 와이파이라도 서로를 못 봅니다.
- RMW = rmw_fastrtps_cpp — DDS 구현이 다르면 발견(discovery) 단계부터 실패합니다.
- 발견 모드 = multicast(SUBNET) — Discovery Server를 섞어 쓰면 한쪽만 달라도 끊깁니다.

## ③ 로봇과 센서

**1. 표지 — 센서 신호는 어떻게 화면까지 오나?**
Arduino에 꽂은 센서 하나의 신호가 ROS 메시지로 바뀌어 Unity 화면에 카드로 나타나기까지의 여정입니다. Arduino Uno, pyserial, `/security/event`, 정본 v18 기준입니다.

**2. 센서 라인업 — 현재와 바뀔 예정**
정본 v18(2026-06-16)에서 조도·불꽃이 빠지고 온도·레이저·워터펌프가 들어왔습니다. 뱃지가 지금 상태입니다.
- PIR(HC-SR501, D2) — 사람 움직임 감지, 현재 사용 중.
- 소형 사운드(KY-038, D3) — 큰 소리 감지로 이상 알람, 현재 사용 중.
- 레이저 송신 모듈(D8) — PIR이 감지할 때 함께 켜져 표시, 이미 달았음.
- 온도 센서 모듈(A0) — 화재·과열 보조 감지, 현재 달려 있음.
- 워터 펌프(릴레이, D5) — 화재 감지 시 물 분사, 새로 달 예정.
- 조도(LDR)·불꽃(Flame) — 정본 v18부터 사용 중단.

**3. 모든 센서가 가는 같은 길 (다이어그램)**
어떤 센서든 같은 경로로 신호가 오갑니다. 센서 4종(PIR·사운드·온도·레이저) → 미니 브레드보드(5V·GND 공통) → Arduino Uno(핀 신호를 시리얼로) → USB-B로 라즈베리파이(115200) → sensor_bridge(시리얼 해석) → `/security/event` 발행 → Unity(지도 마커·센서 카드). 전원은 OpenCR 5V에서 점퍼 2줄로 Arduino 5V로 받습니다. 별도 배터리는 없습니다.

**4. Arduino 한 줄이 화면 이벤트가 되기까지 (flow)**
Arduino가 보내는 글자들이 어떻게 변환되는지 봅니다. PIR은 D2 HIGH → `EVT,pir,3,<ts>` → bridge가 로봇 ID 추가 → `/security/event`(위험도 3) → 지도 마커 + PIR 카드 빨강. 사운드는 D3 HIGH → `EVT,noise,2,<ts>` → 소음 이벤트 → 이상 소음 알림. 온도는 A0 수치 높음 → `EVT,fire,3,<ts>` → 워터펌프 시작 → 화재 경보 화면.

**5. 핀맵 (정본 v18)**
센서가 Arduino 어디에 꽂혀 있는지입니다. 통신속도는 115200 고정. PIR=D2(`pir`, 사용), 소리=D3(`noise`, 사용), 레이저=D8(PIR과 함께, 이미 달음), 온도=A0(`fire`, 사용), 워터펌프=D5(곧). 조도(LDR, A0)와 불꽃(Flame, D4)은 없앴습니다.

**6. 실제 결선 모습 (사진)**
Arduino 하나에 센서 네 종류를 꽂은 실물입니다. PIR·레이저·사운드·온도를 Arduino Uno에 함께 연결했습니다.

**7. 센서가 바뀌면 Unity 화면도 바뀐다**
센서를 빼고 더하면 관제 화면의 센서 카드도 손봐야 합니다. 다음 과제입니다.
- 없앤 것 — LuxSubscriber.cs는 이미 삭제. 남은 건 Unity 조도 카드 제거(dark 이벤트가 안 옴).
- 추가할 것 — 온도 카드와 SensorRegistry 온도 항목, 화재 때 워터펌프 표시.
- 레이저 표시 — PIR 감지 시 레이저 켜지는 모습을 UI 보조 표시로.
- 문서 규칙 — 센서 변경 시 CONTRACT.md §4와 DECISION-LOG를 함께 갱신하고 PR로 요청.

## ④ 관제 화면과 데이터

**1. 표지 — 우리 로봇 데이터는 어디에 모이나**
센서가 감지한 이벤트, 로봇이 움직인 위치, 출동 명령, 카메라 확인 결과를 클라우드(Supabase의 Postgres)에 저장했다가 발표 성과로 정리합니다. 도쿄 서버(ap-northeast-1), Unity에서 HTTP로 직접 저장, 익명 키와 행 단위 보안을 씁니다.

**2. 이 화면의 정보가 그대로 저장됩니다 (사진)**
관제 화면에 보이는 로봇 위치·로그·출동이 곧 데이터베이스 한 줄입니다.

**3. 한눈에 보기 — 6개 테이블 저장 현황**
뱃지는 실제 적용 상태입니다. 적용 완료는 ✅, 복구 후 진행은 ⏳.
- session_meta — 시연 1회 정보, 모든 기록의 부모. (적용)
- events — 센서 감지 이벤트(침입·소음·온도·화재). (적용)
- dispatches — 로봇 출동 명령과 도착 시간. (적용)
- camera_captures — 카메라 확인 결과(맞음/틀림). (적용)
- pose_logs — 로봇 이동 위치, 초당 2회 저장. (적용)
- logs — 관제 화면 로그 라인(화면=DB 일치 확인용). (복구 후 적용)

**4. 테이블 관계도 — 부모와 자식 연결**
한 시연 아래 모든 기록이 달려 있습니다. session_meta가 부모고, 그 아래 events → dispatches → camera_captures로 연결됩니다(센서 감지 → 출동 → 카메라 확인). pose_logs(위치)와 logs(화면 로그)는 시연 바로 아래에 붙습니다. 출동은 이벤트마다 0~1개(손으로 명령하면 이벤트 없음), 카메라 확인은 출동마다 0~1개입니다.

**5. 중요한 컬럼 — 위치 추적과 화면 기록 중심**
발표의 핵심 둘입니다. 로봇 이동 경로(pose_logs)와 화면 로그-DB 일치(logs). 좌표는 미터, 각도는 라디안. pose_logs의 x,y,theta는 맵 위 위치·방향, nav_mode는 순찰/출동 구분. dispatches의 target_x,y는 클릭 좌표, response_time은 출동 응답시간(성과 지표). logs의 level/category/message는 복구 후 적용 대기입니다.

**6. 데이터 저장 흐름 — "직접 저장" 방식 (flow)**
Unity가 HTTP로 데이터베이스에 직접 저장합니다. 로봇 명령(ROS)과 기록(DB)이 별개로 움직입니다. 맵 클릭 출동은 우클릭 좌표 선택 → `/goal_pose` 발행으로 로봇 이동 → dispatches 저장 → 익명 키로 권한 확인. 위치 기록은 ROS 위치 수신 → 초당 2회·최소 이동만 → pose_logs 저장 → 맵에 궤적. 화면 로그는 OnLogAdded → logs 저장 → 화면 로그 수 = DB 로그 수 검증.

**7. 설계 원칙 — 발표 중 끊김 0 + 안전**
발표가 멈추면 안 되고, 중요한 키가 노출되면 안 됩니다.
- 비차단 저장 — 모든 저장을 대기줄에 넣고 바로 반환. 네트워크 지연이 프레임을 막지 않습니다.
- 오프라인 보호 — DB 다운·와이파이 끊김 시 디스크에 모아뒀다가 복구되면 순서대로 자동 전송. 유실 없음.
- 저장 빈도 조절 — 위치는 초당 2회, 실제 움직였을 때만. 요청 폭증 방지.
- 권한 분리(행 단위 보안) — 모바일은 공개 키만, 관리 키는 로봇 컴퓨터에만. 테이블마다 권한 제어.

**8. 발표용 성과 지표 (테이블)**
DB 테이블을 묶어 시나리오별 성능을 한 번에 계산합니다(테스트 값). 침입 감지 12회·4.2초·92%, 소음 7회·3.8초·86%, 화재 3회·5.1초·100%, 야간 순찰 21회입니다.

**9. 데이터베이스 구축 진행률**
구조를 설계하고 5개 테이블을 적용했습니다. 구조 설계는 완료(6/6), 테이블 적용은 5/6(로그 1개 남음), Unity 직접 저장 코드·빌드는 완료(3/3), 로그 테이블과 익명 키 권한은 Supabase 복구 후 대기, 발표 환경 화면=DB 일치 확인도 대기입니다.

## ⑤ 시행착오와 배운 점

**1. 표지 — 쉽지 않았다, 우리가 부딪힌 벽들**
처음부터 잘 된 건 거의 없습니다. 막히고, 고치고, 방향을 바꾼 기록입니다. 2026년 6월 개발 일지와 회의록 정리입니다.

**2. 연결과 지도에서 막힌 것**
로봇을 움직이기도 전에 통신과 지도부터 애를 먹였습니다.
- 와이파이가 자꾸 끊김 — 무선 접속이 배터리 낮을 때 끊기고 불안정. → 맥과 로봇을 직접 선으로 연결해 우회.
- 지도가 깨짐 — 라이다로 지도를 그리는데 제자리에서 돌면 자주 망가짐. → 돌지 말고 늘 앞으로 가며 스캔.
- 주행 값 맞추기 — 자동 주행 값을 잘못 잡으면 길을 못 찾거나 멀리 돎. → 맵이 작아 DWB 방식 선택 + 값 재조정.

**3. 카메라와 인식에서 막힌 것**
보는 것도 생각보다 까다로웠습니다.
- 영상이 뚝뚝 끊김 — RealSense 고해상도는 데이터가 커서 끊김. → 424×240·15fps로 낮춰 부드럽게.
- 학습 사진 부족 — 각도·조명이 적어 물체를 잘 못 맞힘. → 여러 각도로 300장 넘게 다시 수집 중.
- 좌표가 안 맞음 — 화면 속 로봇 위치가 실제와 어긋남. → 출발점과 지도 원점을 손으로 맞춤.

**4. 이미지 인식(YOLO) — 지금 학습 중 (사진)**
직접 모은 데이터로 화재를 탐지하는 중입니다. 화면처럼 `FIRE` 박스가 잡히기 시작했지만 신뢰도가 0.26~0.35로 아직 낮습니다. 학습 초기라 그렇고, 각도·조명을 늘려 데이터를 더 모으는 중입니다. 앞 슬라이드의 "학습 사진 부족 → 300장 재수집"과 이어지는 현재 상태입니다.

**5. 욕심을 줄인 결정들**
다 하려다 일정이 막힐 것 같아 무거운 건 과감히 뺐습니다.
- 자동 충전 포기 — 도킹 접점 정밀도·복잡도가 큼. → ArUco 마커로 주차까지만, 충전은 사람이 연결.
- 무선 충전 접음 — 배터리 호환·회로·일정이 안 맞음. → 유선 충전으로.
- ROS2 재설치 — 젠지의 ROS2가 지워지는 사고. → 다시 설치하고 설정 복구.

**6. 디버깅의 흔적 (사진)**
막힌 순간들을 사진으로 남겼습니다.

**7. 그래서 배운 것**
고생 끝에 남은 네 가지입니다.
- 작게 시작하기 — 완벽한 자동화보다 되는 것부터 확인하고 늘렸습니다.
- 직접 해보기 — 문서보다 실제로 돌려본 결과가 늘 더 정확했습니다.
- 욕심 줄이기 — 자동 충전처럼 무거운 건 과감히 범위에서 뺐습니다.
- 기록 남기기 — 막힌 것과 해결을 회의록에 적어두니 다음이 빨랐습니다.

## ⑦ (부록) Unity 코드 구조

**1. 표지 — Unity 관제실은 어떻게 구성돼 있나**
unity/ControlRoom 프로젝트의 폴더·파일 지도입니다. 어디를 열면 무엇이 나오는지 한눈에. UI Toolkit, ROS-TCP-Connector, Supabase, C# 기반.

**2. 먼저 알아둘 3가지**
- 위치 — unity/ControlRoom/, 씬은 Scenes/ControlRoomMain.unity 하나.
- 화면은 UI Toolkit으로 — UXML로 배치, USS로 꾸밈. Assets/UI/가 정본.
- 코드는 일 별로 — 통신·상태·화면·지도가 폴더로 나뉨. Scripts/ 안에 정리.

**3. Assets 폴더 트리**
Scenes(메인 씬), UI(UI Toolkit 정본 — uxml 레이아웃, uss 스타일·토큰, Parts 5조각), Scripts(App·Data·UI·Map·Ros·Sensors/Features·Database·Simulation·Design), Editor(자동화), Art/IconsPng, Robots(URDF), Resources(런타임 설정), Tests(스모크 테스트)로 구성됩니다.

**4. Scripts 도메인 — 폴더가 곧 책임**
한 폴더 = 한 가지 일. 어디를 고칠지 헷갈릴 때 보는 표입니다. App(실행 시작·선택 로봇/모드 기억·연결), Ros(메시지 받기/보내기, 이름은 TopicRegistry 한 곳에), UI(상태값으로 화면 그리기), Map(2D 지도·점·우클릭 출동), Data(로봇·센서·목표 정보, 불변), Database(Supabase 위치 기록 저장/불러오기).

**5. Scripts/Ros — 통신의 심장**
토픽 1종 = 파일 1개. 새 센서가 생기면 여기 Subscriber가 하나 늘어납니다. TopicRegistry.cs(모든 이름 한군데, 복사 금지), RobotPoseSubscriber, MapSubscriber, CameraStreamSubscriber, BatterySubscriber, PirSubscriber, SoundSubscriber, TemperatureSubscriber, LaserSubscriber(미결선), DispatchPublisher(클릭 시 명령 발행). 조도(LuxSubscriber)는 폐기됨.

**6. 화면 한 장 = UXML Part 5개**
Assets/UI/Parts/ 안의 5조각 조합입니다. TopBar(로봇 고르기·시간·경보·전원), LeftControlPanel(수동 이동·순찰·패턴), MapPanel(가운데 2D/3D 지도, 로봇·사건 표시), CameraAndLogPanel(카메라 영상·사건 기록), RightStatusPanel(배터리·센서 상태·기능 토글).

**7. 새 센서/기능을 추가하려면? (수정 순서)**
예로 온도 센서를 새로 붙일 때 손대는 곳입니다. 1) TopicRegistry.cs에 이름 등록 → 2) Ros/○○Subscriber.cs로 값 받는 코드 → 3) App/ControlRoomState와 Events에 기억·전파 → 4) UI/SensorCardListView와 Sensors/SensorRegistry에 화면 박스.
